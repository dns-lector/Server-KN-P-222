Вступ до Django
Django - Python фреймворк для веб-розробки.
Перед початком роботи пересвідчуємось у наявності python та його пакетного менеджера pip
PS C:\...> pip --version
pip 25.2 from C:\Program Files\Python313\Lib\site-packages\pip (python 3.13)
Якщо програма не виконується, то
а) перевіряємо чи встановлений Python і чи є в нього pip. Якщо є, то потрібно або додати
    його шлях до системної змінної PATH, або зазначати повний шлях до pip при його запуску.
б) якщо Python немає, то встановлюємо його

Створюємо віртуальне оточення Python
Особливістю Python є те, що він створювався як єдина програма в операційній системі і тому
всі модулі, що встановлюються додатково, суть актуальні для всіх проєктів користувачів.
Для забезпечення можливості функціонування різних проєктів окремо, прийнято створювати 
віртуальні оточення.
Переходимо у директорію, вищу за корень проєкту та вводимо команду
> python -m venv django
З'являється нова директорія django, яку відкриваємо як проєкт у редакторі (VSCode, PyCharm, ...)

------------------ При кожній новій роботі з проєктом -----------------------------------------

Активуємо віртуальне оточення:
для PowerShell:
> ./Scripts/Activate.ps1
для Command Prompt
> ./Scripts/activate.bat
Ознакою активації є зміна промпту із зазначенням назви оточення
(django) ...>

------------------------------- 
Встановлюємо Django
python -m pip install Django

після встановлення додається скрипт django-admin.exe через який 
буде здійснюватись подальше налаштування. Можна перевірити його роботу 
(успішність встановлення фреймворка)
(django)> django-admin --version
6.0.3

Структура Django складається з проєктів та застосунків, для роботи 
необхідний принаймні один проєкт, у якому є принаймні один застосунок
(django)> django-admin startproject project_222
Додається директорія project_222 з головним скриптом управління 
проєктом - manage.py - через який продовжується робота. Також 
створюється база даних (db.sqlite3)

Переходимо в директорію проєкта та запускаємо сервер
(django)> cd project_222
(django)> python manage.py runserver
Зупиняємо сервер (Ctrl-C)
і створюємо застосунок 
(django)> python manage.py startapp app222

Архітектура
MVT - Model View Template
Model - постачальник даних
Template - шаблони відображення - заготовки веб-сторінок з підстановкою даних
View - функції, що відповідають за підготовку даних для шаблонів (Template)
       Цей модуль аналогічний контролерам у MVC

       
Реалізуємо маршрутизацію.
app222/views.py - заходимо, редагуємо - створюємо функцію представлення

Створюємо файл внутрішньої маршрутизації: app222/views.py (див. файл)
Додаємо цей файл до проєктної маршрутизації: див. project_222/project_222/urls.py 
(в ньому є інструкції - коментарі "з коробки", діємо за п."Including another URLconf")

--------------- Template --------------------------
Шаблони - HTML-сторінки з потенційною можливістю підстановки даних
створюємо директорію app222/templates
у ній створюємо файл home.html
У файлі app222/views створємо функцію, що рендерить даний шаблон (див. файл)
Додаємо маршрут у файлі app222/urls.py

Реєструємо застосунок у налаштуваннях проєкту project_222/settings.py :
INSTALLED_APPS = [
    ...
    'app222'   --- додаємо цей рядок
]

Включаємо міграції
(django)> python manage.py migrate
(django)> python manage.py runserver

------------------ Контекст та передача даних до шаблону ---------------------
Контекст - dict, який передається від View до Template і автоматично
перетворюється у локальні змінні шаблону з іменами, що відповідають 
ключам контексту. Якщо потрібна інформація про весь контекст, то її 
слід також включити до самого контексту (окремих засобів немає)

------------------ Запуск проєкту після перезапуску ---------------
- Переходимо до директорії проєкту  `cd django`
- Активуємо віртуальне оточення, для PS команда 
    `./Scripts/Activate.ps1`, переконуємось у зміні командного промпту 
    (додається (django) на початку промпту)
- Переходимо до Django-проєкту `cd project_222`
- Запускаємо сервер проєкту `python manage.py runserver`, 
    переходимо до браузеру

------------------ Layouting - шаблонізація представлень -------------
На відміну від більш поширених практик включення представлень у шаблон (Layout), 
у Django вживається принцип переозначення (спадкування) шаблону за допомогою
іменованих блоків

   Layout                             Page 
<header>                            {% extends 'Layout' %}
{% block name_of_block %}           {% block name_of_block %}             
Default Content                      The content of block
{% endblock %}                      {% endblock %}  
<footer>                            <p>Ignored content</p>

Шаблони, які переозначають інші шаблони, можуть лише перезаповнювати блоки.
Якщо блок не зазначається, то вживається значення від попереднього шаблону.
Якщо додається контент поза блоками, то він ігнорується. 
Контекст, переданий до сторінки, зберігається при використанні шаблону. 


--------------- Статичні файли -----------------------------
Знаходимо файл конфігурації settings.py і дивимось посилання на інструкції 
(близько до кінця файлу)
https://docs.djangoproject.com/en/6.0/howto/static-files/
{% load static %}  -- треба додати до сторінок, які посилаються на статичні
файли, зокрема layout.html
Самі посилання на файли слід зазначати як href="{% static '/css/site.css' %}"
Також до конфігурації (settings.py) слід додати інструкції:
STATICFILES_DIRS = [
    BASE_DIR / "static",
    "/var/www/static/",
]
