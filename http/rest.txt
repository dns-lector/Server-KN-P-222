REST - Representational State Transfer
Класично - набір архітектурних вимог (обмежень) на створення бек-АРІ
Еволіційно - погляд на стуктуру даних, що передається по АРІ

Метадані та їх роль
Метадані - дані, що супроводжують основні дані, не є їх частиною (дані про дані)
приклади:
 - надписи-номера на перфокартах
 - імена файлів (у т.ч. директорій - повні імена)
 - Бази Даних та мова DDL - метадані "змішуються" з самими даними

Сприйматимемо REST як намір впровадити до відповідей АРІ метаданих:
 - cacheability - відомості про час повторного використання даних без перезапиту
 - Resource identification - включити відомості про URI даного ресурсу (джерела даних)
 - Resource manipulation - відомості про модифікацію (у т.ч. видалення) ресурсу
 - Self-description - відомості про тип даних, спосіб його перетворення чи оброблення
 - HATEOAS - відомості про підлеглі, внутрішні розділи даного розділу, зміст розділу
 + (за аналогією Swagger) - опис структури даних
 + в залежності від типів даних (для масивів) - пагінація (номер сторінки, місткість сторінки
    загальна кількість елементів, загальна кількість сторінок/остання сторінка, 
    посилання на [попередню, наступну, першу, останню сторінки])

? Шукаємо місце для метаданих. Перше, що спадає на думку - НТТР-заголовки
! Корегування вносить вимога "Layered system" - клієнт не повинен розрізняти 
  запит прямо до сервера або до транзитних вузів (проксі)
- Кожен проміжний сервер може (і навіть повинен) змінювати деякі заголовки:
 = мітку часу оброблення даних
 = свою назву як сервера

Client ------- [Proxy Apache]----------[Server Tomcat]

        GET                       GET
запит ---------->             -------------> |
                                             |
     Server:Apache             Server:Tomcat |
    <------------- Транзит    <------------- |
     time:123120003            time:123120001


        GET /x                    GET /x
запит ---------->             -------------> |
                                             |
          507                       404      |
    <------------- Транзит    <------------- |
   Insufficient Storage          Not Found
     
-------------------- Висновок -----------------------
Заголовки не годяться для передачі метаданих за умови, 
що передбачається можливість багатошарових (транзитних)
схем передачі даних.
Вживаємо підхід, аналогічний до БД - "змішуємо" дані 
та метадані в одній структурі

З урахуванням висновків формуємо зразок шаблону відповіді
{
    meta: {
        service: "Back API service 'My shop': products",
        serviceUrl: "https://my-shop.ua/product",
        stateUrl: "https://my-shop.ua/product?code=logi-123",
        method: "GET",
        serverTime:  1234123575,
        cache: 86400,
        manipulation: {
            "create": "POST https://my-shop.ua/product",
            "read": "GET https://my-shop.ua/product?code=logi-123",
            "update": "PATCH https://my-shop.ua/product?code=logi-123",
            "delete": "DELETE https://my-shop.ua/product?code=logi-123"
        },
        links: {
            "producer": "https://my-shop.ua/product/logi",
            "group": "https://my-shop.ua/product/mouse",
            "category": "https://my-shop.ua/product/computer"
        },
        dataType: "object",
        dataModel: {   // або посилання на модель https://my-shop.ua/model/product
            id: {
                fieldType: "string",
                fieldSubType: "UUID",
                fieldDescription: "Product identifier"
            },
            ...
            price: {
                fieldType: "number",
                fieldSubType: "float",
                fieldDescription: "Product price in UAH"
            }
        }
    },
    data: {
        id: "123f1a-...",
        code: "logi-123",
        name: "Mouse",
        price: 250
    }
}
